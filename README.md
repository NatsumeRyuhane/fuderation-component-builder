# Fuderation Workshop Component — Template

A template for building **one** [Fuderation Workshop](https://chat.fuderation.com/guide#component-guide-section-1)
component — an interactive UI widget that renders inline in a chat message in a
storyline. You author source in `src/`; the build produces a single
`component.json` at the repo root that you import directly into Workshop.

> One component per repo. Use this template once per component you build.

## Prerequisites

- **Node.js 18+** (developed on Node 22) and npm.
- Dev dependencies, installed with `npm install`:
  - **esbuild** — compiles `src/script.ts` to a minified inline script.
  - **typescript** — type-checks `src/script.ts` against the bridge-function
    declarations in `types/bridge.d.ts`.

No runtime/production dependencies — the component ships as plain HTML/CSS/JS.

## Quick start

```bash
npm install        # install esbuild + typescript
# edit the files in src/ (start from the info-card scaffold)
npm run build      # writes ./component.json
```

`component.json` is **generated by the build and is gitignored** — it does not
exist in a fresh clone and won't appear until you run `npm run build`. Re-run the
build after every change to `src/`.

Then in Workshop: open your storyline → **Components** → import `component.json`.
Save the storyline and **playtest** — preview alone does not prove the AI will
invoke the component.

## Project layout

```text
.
├── src/                 # what you author
│   ├── markup.html      # HTML only            -> component.html
│   ├── styles.css       # styles               -> component.css
│   ├── script.ts        # compiled (esbuild)   -> component.script   (iframe mode)
│   ├── script.js        # OR verbatim          -> component.script   (DSL mode)
│   ├── ai_prompt.md     # AI supplementary prompt -> component.ai_prompt
│   └── meta.json        # { "name", "description" }
├── component.json       # BUILD OUTPUT (generated, gitignored) — import this into Workshop
├── scripts/build.mjs    # the builder
├── types/bridge.d.ts    # ambient types for the bridge functions
└── .agents/skills/      # the authoring skills (workflow + design + reference)
```

`$Param$` placeholders in `markup.html`/`styles.css`/`script.*` are filled by the
AI at invocation time via `<$name$><Param>value</Param></$name$>`.

## Authoring the script

Use **either** `src/script.js` **or** `src/script.ts` — not both.

- **`script.js` (verbatim, recommended for simple components)** — write one
  bridge call per line, e.g. `setText('[data-out]', '$Text$')`. It is passed
  through untouched, so it runs in the runtime's lightweight **DSL mode**.
- **`script.ts` (compiled)** — bundled and minified to an inline IIFE by esbuild.
  Any compiled output runs in the sandboxed **iframe mode**. Write normal
  TypeScript; **do not `import` the bridge functions** — they are injected as
  globals and declared ambiently in `types/bridge.d.ts`. No external/CDN scripts
  and no networking are allowed inside the iframe.

Type-check TypeScript sources with `npm run typecheck`.

## Constraints (enforced by the build)

- `name` ≤ 32 chars; letters/digits/`-`/`_`/CJK only.
- `description` ≤ 120 chars; `ai_prompt` ≤ 1000 chars.
- `html` + `css` + `script` ≤ 20000 chars combined.
- No real networking, auth, or payment. Components are disabled in VN mode.
